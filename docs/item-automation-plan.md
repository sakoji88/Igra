# План автоматизации предметов и событий

## Что уже есть в коде
- Автоматически разрешаются только простые числовые эффекты движения и лок на выбор условий через `resolveRollEffects` и `resolveConditionSelectionEffects`.
- Вручную помечаются почти все остальные предметы через `manualNote`, то есть движок пока не умеет держать долгие состояния, модифицировать временные окна, управлять рероллами и обрабатывать cross-player эффекты.

## Предлагаемая модель внедрения

### 1. Разделить все эффекты на 5 механических классов
1. **Numeric auto** — движение, поинты, выбор base/genre, бонус за круг, фильтры колеса.
2. **State auto** — эффекты с длительностью: `next_roll`, `next_2_runs`, `until_board_lap`, `while_active`, `until_jail_resolved`.
3. **Target auto** — эффекты, которые вешаются на другого игрока и снимаются после точного триггера.
4. **Choice auto** — автоматическая механика с серверным выбором из ограниченного набора (`выбери соседний сектор`, `выбери 2 из 3 кубов`, `замени сектор`, `сбрось дебафф или бафф`).
5. **Judge/manual hybrid** — вещи с внешним источником истины: чат-голосование, квизы, Discord, судья, ручное подтверждение.

### 2. Какие сущности добавить
- `PlayerEffectState`: активные эффекты с полями `effectCode`, `sourceItemNumber`, `durationType`, `remainingUses`, `payloadJson`, `ownerStateId`, `targetStateId`, `expiresAt`, `consumedAt`.
- `BoardCellModifier`: временные изменения конкретной клетки (`timeLowerDelta`, `timeUpperDelta`, `forceNoUpperBound`, `forceGenreOnly`, `createdByStateId`, `expiresOnUse`).
- `WheelSpinContext`: контекст реролла/выбора соседей/бафф-колеса/дебафф-колеса.
- `ScoreTransferLink`: связки вроде `Рука для fisting` или `Парные кольца времени`.

### 3. Какие новые типы эффектов нужны движку
- `die_override`, `largest_die_set`, `roll_mode`, `move_multiplier`, `move_backwards`, `double_bonus`, `score_on_double`
- `time_window_delta`, `time_window_remove_upper`, `time_window_force_minimum`, `tag_filter_add`, `tag_filter_remove`
- `wheel_pool_override`, `wheel_neighbor_choice`, `wheel_reroll`, `wheel_sector_replace`, `wheel_vote_choice`
- `inventory_destroy_random`, `inventory_destroy_by_type`, `inventory_swap`, `inventory_charge_add`, `trap_reflect`
- `drop_redirect`, `drop_score_penalty`, `start_bonus_override`, `score_transfer_link`
- `game_picker_mode`, `release_date_filter`, `price_filter`, `difficulty_lock`, `guide_permission`

## Что я бы изменил в коде прямо по архитектуре
1. Перестать хранить механику предметов как почти сплошные `manualNote` и перейти к декларативным effect-конфигам для всех предметов.
2. Расширить `effect-engine.ts`, чтобы он умел не только `move_modifier` и `condition_lock`, а ещё long-lived effect states и server choices.
3. Добавить единый pipeline хода: `before_roll -> roll_mode -> after_roll -> before_move -> move -> after_move -> before_assignment -> assignment -> active_run -> score -> cleanup`.
4. Завести отдельный слой `resolveDropEffects`, `resolveWheelEffects`, `resolveTimeWindowEffects`, `resolveInventoryDestructionEffects`, `resolveScoreTransferEffects`.
5. Все предметы с таргетом перевести на серверные транзакции, чтобы владелец и цель менялись атомарно.
